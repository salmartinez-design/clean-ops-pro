PART A — API SECURITY AUDIT

Go through every single API route in the codebase and verify:

1. AUTHENTICATION CHECK
Every route except /api/auth/login and /api/auth/register 
must have authentication middleware that:
- Validates the JWT token
- Returns 401 if token is missing or invalid
- Returns 401 if token is expired
List any routes missing this check and add it.

2. COMPANY_ID ISOLATION
Every route that reads or writes data must:
- Extract company_id from the JWT payload (not from the 
  request body or URL params)
- Filter ALL database queries by company_id
- Never trust company_id sent from the client
- Return 403 if the requested resource belongs to a 
  different company_id

Go through every single GET, POST, PUT, PATCH, DELETE 
route and verify this. Fix any that are missing it.
List every route you check and its status.

3. ROLE AUTHORIZATION
Verify these role restrictions are enforced server-side 
on every route (not just client-side):
- Super admin routes (/api/admin/*): role must = 
  'super_admin'
- Owner/admin routes (payroll, close day, batch invoice, 
  company settings): role must be 'owner' or 'admin'
- Office routes (create jobs, invoices, quotes): role must 
  be 'owner', 'admin', or 'office'
- Technician routes (/api/my-jobs, clock in/out, photos): 
  any authenticated role
Fix any routes where role is checked client-side only.

4. INPUT VALIDATION
Add server-side input validation to every POST and PUT 
route. For each route:
- Validate required fields are present and not empty
- Validate data types (numbers are numbers, dates are dates)
- Validate string lengths (names max 100 chars, notes 
  max 2000 chars)
- Sanitize inputs — strip HTML tags from all text fields
- Return 400 with specific error message for invalid inputs
Never trust client-side validation alone.

5. SQL INJECTION PROTECTION
Verify all database queries use parameterized queries 
or Supabase's query builder (never string concatenation).
Flag and fix any raw SQL with string interpolation.

6. SENSITIVE DATA
Ensure these fields are NEVER returned in API responses:
- Password hashes
- JWT secrets
- SSN last 4 (only return masked: ***-**-XXXX)
- Bank account numbers (only return last 4)
- Stripe payment method details beyond last 4 digits
Add a response sanitizer that strips these fields.

PART B — RATE LIMITING

Install and configure express-rate-limit:
npm install express-rate-limit

Apply these rate limits:

1. Authentication endpoints (/api/auth/*):
   - 10 requests per 15 minutes per IP
   - On limit: return 429 with message "Too many login 
     attempts. Please try again in 15 minutes."
   - This prevents brute force attacks

2. General API endpoints:
   - 300 requests per minute per authenticated user
   - On limit: return 429 with message "Rate limit 
     exceeded. Please slow down."

3. File upload endpoints:
   - 20 requests per hour per user
   - On limit: return 429

4. SMS/Email sending endpoints:
   - 50 requests per hour per company
   - On limit: return 429 with message "Message limit 
     reached for this hour."

Log all rate limit hits to a rate_limit_log table:
  id, ip_address, endpoint, user_id, company_id, 
  hit_at (timestamptz)

PART C — ERROR HANDLING

1. GLOBAL ERROR HANDLER
Add a global error handler middleware in Express that:
- Catches all unhandled errors
- Logs the full error to console with timestamp, 
  endpoint, user_id, company_id
- Returns a clean error response to the client:
  { error: "Something went wrong. Please try again." }
- Never exposes stack traces or internal error details 
  to the client in production
- For development: include error details
  Check NODE_ENV to determine which to use

2. ASYNC ERROR WRAPPER
Wrap every async route handler with a try-catch wrapper 
so unhandled promise rejections don't crash the server.
Create a utility: asyncHandler(fn) that wraps any async 
route function and passes errors to next().
Apply it to every route.

3. 404 HANDLER
Add a catch-all 404 handler at the end of all routes:
- Returns { error: "Route not found" } with 404 status
- Logs the attempted route and IP to console

4. DATABASE ERROR HANDLING
For all Supabase queries:
- If query returns an error, log it server-side with 
  context (what operation, what table, company_id)
- Return appropriate HTTP status to client:
  - Unique constraint violation: 409 Conflict
  - Not found: 404
  - Permission denied: 403
  - All others: 500

5. CLIENT-SIDE ERROR BOUNDARY
Add a React error boundary component that wraps the 
entire app. When a component crashes:
- Show a clean error screen: "Something went wrong. 
  Please refresh the page."
- Include a "Refresh" button
- Log the error to console
- Do NOT show technical error details to the user

  id, company_id, performed_by (user_id), action (text),
  target_type (text), target_id (text), 
  old_value (JSONB), new_value (JSONB),
  ip_address (text), user_agent (text),
  performed_at (timestamptz)

Log these actions automatically:

- User login (success and failure)
- User logout
- Password change
- Role change for any user
- Employee created or deleted
- Client created or deleted
- Invoice created, sent, paid, deleted
- Payment recorded or reversed
- Job created, completed, deleted

- Any super admin action (impersonation, company edit)

- Any bulk action (batch invoice, bulk delete)

MEDIUM PRIORITY:

- Who did it (user id + name)
- What they did (action string)
- What was affected (target_type + target_id)
- Before and after values for edits (old_value/new_value)
- When (timestamp)
- Where from (IP address)

PART F — SESSION SECURITY

1. JWT CONFIGURATION
Verify JWT tokens are configured with:
- Expiry: 24 hours (not longer)
- Algorithm: HS256
- Secret: minimum 32 characters (check JWT_SECRET length)
- Payload contains: user_id, company_id, role, email
- Payload does NOT contain: password, sensitive data

2. TOKEN REFRESH
Add a token refresh endpoint: POST /api/auth/refresh
- Accepts a valid (not expired) JWT
- Returns a new JWT with fresh 24-hour expiry
- Call this automatically on the client when token 
  has less than 2 hours remaining
- If token is already expired: return 401, redirect to login

3. LOGOUT
Ensure logout on the client:
- Clears the JWT from localStorage/sessionStorage
- Clears any cached user data from React state
- Redirects to /login
- The server-side logout endpoint should log the 
  logout event to audit_log

4. CONCURRENT SESSION HANDLING
For now (MVP): allow multiple sessions per user.
Log each login to audit_log with device/browser info.
This gives visibility into suspicious logins.

PART G — FRONTEND SECURITY

1. XSS PREVENTION
Verify all user-generated content displayed in the UI 
is properly escaped. Never use dangerouslySetInnerHTML 
unless absolutely necessary. If used anywhere, audit it.

2. SENSITIVE DATA IN FRONTEND
Ensure these are NEVER stored in localStorage:
- Passwords
- Full payment card numbers
- SSN data
The JWT token in localStorage is acceptable for MVP.

3. ENVIRONMENT VARIABLES
Verify no secret keys (STRIPE_SECRET_KEY, 
RESEND_API_KEY, SUPABASE_SERVICE_KEY) are exposed 
to the frontend bundle. Only STRIPE_PUBLISHABLE_KEY 
and SUPABASE_ANON_KEY should be accessible client-side.
