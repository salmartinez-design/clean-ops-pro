FEATURE: Smart Dispatch Suggestion

When a job is being scheduled (during booking creation or editing a job's 
assigned technician), the system should automatically suggest the best 
available technician based on:

1. AVAILABILITY — Filter out any technician who has an overlapping job 
   scheduled within a 30-minute buffer on either side of the new job's 
   start/end time.

2. PROXIMITY — Using the existing Service Zones (zip code based, already 
   built), rank available techs by how close their zone is to the new 
   job's zip code. Since Google Maps is not yet configured, use zip code 
   zone matching as the proximity proxy:
   - Tier 1: Tech's assigned zone MATCHES the job's zip code exactly
   - Tier 2: Tech's last job of the day is in a zone adjacent/same as 
     the new job's zip
   - Tier 3: Tech's home zip is in the same zone as the job
   - Tier 4: All other available techs

BACKEND — Add a new API endpoint:
POST /api/jobs/suggest-tech
Request body: { date, start_time, end_time, zip_code, company_id }

Logic:
1. Query all active technicians for the company
2. Query jobs table for that date to find conflicts (overlap = 
   start_time - 30min to end_time + 30min)
3. Remove conflicted techs from the pool
4. Score remaining techs by proximity tier (1 = best)
5. Return ranked array: [{ employee_id, name, tier, reason, 
   zone_color, zone_name, last_job_end_time }]

FRONTEND — In the job scheduling workflow (job creation modal or 
the tech assignment step), add a "Suggested Techs" section that 
appears immediately after date/time and address are filled in:

UI BEHAVIOR:
- Trigger the suggestion API call automatically when date, 
  start_time, end_time, AND zip_code are all populated
- Show a subtle loading state while fetching (skeleton rows, 
  not a spinner)
- Render a ranked list of up to 5 suggestions
- Each row shows:
  - Tech name (primary text)
  - Zone color dot (existing zone color from Service Zones)
  - Proximity tier label (e.g., "Same Zone", "Adjacent Zone", 
    "Available - Different Zone")
  - Availability status (e.g., "Next available at 2:30 PM")
- Highlight the top suggestion with a subtle left border in 
  var(--brand) color
- A "Assign" button on each row that selects that tech and 
  collapses the suggestion panel
- If no suggestions available, show: "No techs available in 
  this window — adjust time or date"

ADDITIONAL RULES:
- This suggestion panel is read-only advisory — dispatcher 
  can still manually override and pick any tech
- Do not block job creation if suggestions fail to load
- Respect existing RLS and company_id scoping — techs from 
  other tenants must never appear
- If a tech has no zone assigned, they appear last in the 
  list with label "No zone assigned"
- The feature should work on both desktop dispatch board 
  AND mobile job creation flow
- No new dependencies — use only what is already in the stack

After implementing, confirm:
1. The suggest-tech endpoint is protected by JWT middleware
2. RLS is respected — company_id filter applied at query level
3. The frontend gracefully handles API errors (silent fallback, 
   no crash)
4. The suggestion refreshes automatically if date/time/zip changes
